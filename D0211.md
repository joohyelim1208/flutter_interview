# 플러터 면접 스터디

## 숙련
- 0211 : Q. MVVM 패턴이란 무엇인가요?
핵심: 계층의 역할과 패턴을 사용하는 이점 
추가하기:

A. MVVM 아키텍쳐는 Model, View, ViewModel의 약자로 앱의 로직과 UI를 분리하여 유지보수성을 높이지 위해 고안된 아키텍쳐 패턴입니다.

각 계층의 역할은
Model 모델 : 앱에서 사용되는 실제 데이터와 데이터를 처리하는 비즈니스 로직을 담당합니다. (예시. Entitiy, API통신 코드)
VIew 뷰 : 사용자에게 보여지는 UI화면. ViewModel로 부터 데이터를 받아 화면에 그리고 사용자의 입력을 ViewModel로 전달합니다. 로직을 직접 처리하지 않음(예시. homepage 클래스)
ViewModel 뷰모델 : View와 Model 사이의 중간다리 역할을 하며 View에 최적화 된 데이터를 가공하여 제공하고, View의 상태를 관리합니다.
'데이터 바인딩'을 통해 view가 ViewModel의 상태를 자동으로 관찰하도록 설계합니다.

데이터바인딩 이란? 
반응형 연결이라고 부르며, ViewModel의 데이터가 변하면 View의 상태가 갱신되기 위해서 build를 실행하도록 자동으로 연결하는데
Notifier, Provider, Bloc같은 상태관리 도구를 사용해서 연결고리를 만듭니다.

MVVM이 작동되는 구조를 설명하자면
사용자가 UI화면에서 버튼 등을 터치했을 때의 입력들이 View를 통해서 들어오고, ViewModel로 전달이 되고 뷰모델에서는 모델에게 데이터를 요청하고
모델은 뷰모델에게 요청받은 데이터를 넘겨주고 뷰모델은 넘겨받은 데이터를 가공해서 저장한 뒤
뷰에서는 데이터 바인딩을 이용해 UI를 갱신시킵니다.ㄴ

MVVM을 사용하는 핵심적인 이유?
1. 관심사를 분리해서 UI 수정 시 로직을 건드릴 필요가 없고 반대의 경우도 마찬가지고 팀프로젝트 시 디자이너와 개발자의 협업에 용이합니다.
2. 테스트의 용이성 : ViewModel은 UI(View)에 의존하지 않기 때문에 화면에 띄우지 않고도 '유닛 테스트를' 진행할 수 있어 안정적인 코드작성이 가능합니다.
3. 코드 재사용성 : 하나의 ViewModel을 여러개의 View에서 공유하거나, 유사한 UI에 같은 Model 로직을 재사용하기 좋습니다.

예시
제가 작업한 TodoList 프로젝트를 예시로 들면, `TodoEntity`가 Model이 되고, 할 일의 목록 UI를 보여주는 `HomePage`가 View가 됩니다.
여기서 `todoList를 추가하고 삭제하는 비즈니스 로직을 별도 클래스로 관리하는게 ViewModel이 됩니다. 
`HomePage`는 View를 어떻게 그릴지에만 집중할 수 있게 됩니다.

추가질문.
1. MVC패턴과 MVVM의 차이란?
MVC는 Controllor가 View와 Model을 연결하지만, View와 Model 사이의 의존성이 완전히 사라지지는 않습니다.
MVVM은 View와 ViewModel이 완전히 분리되어 있으며, View는 ViewModel이 누구인지는 알지만 ViewModel은 View가 누구인지 모르는 상태로 설계되어 의존성이 낮습니다.

2. 패턴을 적용한다는 것이 구체적으로 어떤 행위인가요?
단순히 파일을 나누는 것 이상의 시스템 설계를 말합니다. 
패턴을 적용한다는 것은 하나의 클래스가 감당하던 데이터관리, 비즈니스 로직 처리, UI렌더링 이라는 세가지 책임을 각각 전문 계층으로 나누어서 맡기는 것을 뜻합니다.
